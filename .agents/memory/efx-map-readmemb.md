## Rule

`sapphire.v` initialises the Sapphire SoC on-chip BRAM via four `$readmemb` calls
with bare filenames (no path prefix):

```verilog
$readmemb("EfxSapphireSoc.v_toplevel_system_ramA_logic_ram_symbol0.bin", ram_symbol0);
```

EFX_MAP resolves these relative to its **working directory**, which is `work_syn/`
inside the Efinity project directory — NOT the project root.

**Why:** The symbol files are generated by `make -C firmware` and land in
`hardware/soc_combined/`.  If they are not also copied to `work_syn/`, EFX_MAP
silently zero-initialises the BRAM.  The synthesis still passes (`last_run_state=pass`)
with no warning — the CPU then executes NOPs and the UART never produces output.

**How to apply:** Every time firmware is rebuilt:

```bash
cp hardware/soc_combined/EfxSapphireSoc.v_toplevel_system_ramA_logic_ram_symbol*.bin \
   hardware/soc_combined/work_syn/
# THEN run synthesis
bash hardware/soc_combined/work_syn/run_efx_map.sh
```

Also: `sapphire_define.vh` does NOT contain address defines (it is empty of hex
constants).  UART address (`0xF8010000`) comes from the Efinix BSP `soc.h`.
The `jtagCtrl_*` ports in `sapphire.v` are tied to constants in `top.v`
(debug disabled) — OpenOCD cannot reach the CPU via JTAG.
