CM Lazy Load Protocol
Church Machine · Absent-lump fetch path

Overview
When a thread executes LOAD against a c-list slot whose GT type is Outform (typ=11), the lump binary is not resident in physical memory. The hardware fires the Absent event and hands control to the Locator, which runs as a secure subroutine in the same thread. The lump is fetched, inflated, and validated without any scheduler transfer or thread park visible to the caller. On return from the Locator, the calling thread retries the LOAD against a now- Live NS slot and continues normally.

This document specifies the exact steps of that protocol.

Prerequisites
Term	Definition
Outform GT	A c-list slot Word 0 with typ=11. Signals that the lump is registered but not yet resident. The object_id field identifies which NS slot holds the recovery token.
Outform NS slot	The three NS words for the object hold a 96-bit opaque IDE token (Words 1–3). Hardware passes this token to the Locator when the Absent event fires.
Live NS slot	The three NS words hold the real lump descriptor: base (Word 1), gt_seq + limit_offset (Word 2), CRC-16 (Word 3). LOAD succeeds.
Locator	A ROM-resident or namespace-installed lump invoked as a secure CALL. It owns the fetch-inflate-validate sequence and holds NetworkIO, Mint, and NamespaceWrite capabilities in its c-list.
IDE token	The 96-bit value in the Outform NS slot. Opaque to hardware; interpreted by the Locator (or IDE agent) to identify and authenticate the lump source.
Absent-Lump Protocol — Step by Step
Step 1 — Absent event fires
Actor: Hardware

Trigger: Thread executes LOAD against a c-list slot; GT typ=11 (Outform).

Action:

Hardware reads the three Outform NS words for object_id, extracting the 96-bit IDE token.
Hardware invokes the Locator as a secure subroutine CALL in the same thread. There is no scheduler transfer, no thread park, no context switch. The calling thread's register state is live; the Locator executes on its stack.
The 96-bit IDE token is passed to the Locator via the CALL argument convention (DR registers).
Note: Any concurrency used internally by the Locator (Flag pool, CHANGE) is an implementation detail invisible to the calling thread at this level of the protocol. See LUMP_ARCHITECTURE.md § "Locator and Flag Pool" for the Flag pool design.

Step 2 — Locator preserves the IDE token
Actor: Locator

Input: 96-bit IDE token (passed in DR registers by hardware in Step 1).

Action:

The Locator saves the 96-bit IDE token from the Outform NS slot into its own working memory before any NS slot modification occurs.
This saved token is the restore token: if the lump is later evicted, the NS slot must be reset to the Outform state, which requires writing the original token back into Words 1–3. Without this save, eviction would be unable to restore the lazy-load path.
Output: IDE token preserved for eviction restore path.

Failure: None. Saving is always possible before any network activity.

Step 3 — Fetch the Lump.zip
Actor: Locator (via NetworkIO capability)

Input: 96-bit IDE token (saved from Step 2).

Action:

The Locator resolves the IDE token to a network source (URL, DHT key, or local cache address). URL resolution is out of scope for this document; see LUMP_ARCHITECTURE.md § "The Locator Abstraction".
The Locator fetches the lump zip file. The zip file is the intermediate buffer: the raw compressed bytes are written into a temporary working region that the Locator holds via a NetworkIO-derived RW-GT.
Inflate (Step 5 below) reads from this zip buffer and writes to the separately allocated lump region. There is no second intermediate copy.
Output: Zip bytes available in the Locator's fetch buffer.

Rejection: Fetch failure (network error, timeout, no such resource) → go to Failure: fetch error.

Step 4 — Read ZIP local file header; derive n
Actor: Locator

Input: Zip buffer (from Step 3).

Action:

Read the first ~32 bytes of the zip buffer (the ZIP local file header).

Verify signature = 0x04034B50. Reject on mismatch.

Read byte offset 6 (general-purpose bit flags). Assert bit 3 = 0.

Why bit 3 must be clear: When bit 3 is set (streaming / data descriptor mode), the compressor did not know the output size when writing the header; fields at offsets 16–27 (CRC-32, compressed size, uncompressed size) are all zero. The true values appear in a Data Descriptor record appended after the compressed payload. This defeats streaming pre-allocation: n cannot be derived until the entire stream has been received. Lump zips must be produced with bit 3 clear so the uncompressed size is present in the header.

Read uncompressed_size from byte offset 24.

Derive n = log2(uncompressed_size / 4).

Reject if uncompressed_size is zero or not a power-of-two multiple of 4.
Reject if n < 6 (minimum lump size is 2^6 = 64 words).
Reject if n > 14 (maximum lump size is 2^14 = 16 384 words).
Read the CRC-32 stored at byte offset 16. Save it for Step 6.

Compute the data start offset: 32 + file_name_length + extra_field_length (lengths at byte offsets 28 and 30).

Output: n, saved CRC-32, data start offset.

Rejection: Bad signature, bit 3 set, invalid size → go to Failure: zip format error.

Step 5 — Pre-allocate physical memory
Actor: Locator (via Memory Manager capability)

Input: n (derived in Step 4).

Action:

Call MemoryManager.alloc(n) → receive base (a power-of-two-aligned physical byte address).
The region [base, base + 2^n × 4) is now reserved for this lump. Allocation happens before inflate so inflate can write directly into the destination with no second copy.
Output: base — physical address of the inflated lump region.

Rejection: Allocation failure (out of memory) → go to Failure: allocation error.

Step 6 — Inflate zip payload into lump region
Actor: Locator

Input: Zip buffer (Step 3); base (Step 5); data start offset and compression method (Step 4).

Action:

Seek to the data start offset in the zip buffer.
Decompress (method 0 = STORE: copy directly; method 8 = DEFLATE; method = custom RLE) from the zip buffer into [base, base + 2^n × 4).
Inflate reads from the zip buffer and writes to the pre-allocated lump region. The zip file is the sole intermediate form; no third copy exists.
Output: Lump binary resident at [base, base + 2^n × 4).

Rejection: Decompressor error (corrupt stream) → free the allocated region → go to Failure: zip format error.

Step 7 — Verify zip CRC-32
Actor: Locator

Input: Inflated lump at [base, base + 2^n × 4) (Step 6); stored CRC-32 (Step 4).

Action:

Compute CRC-32 over the inflated bytes now at [base, base + 2^n × 4).
Compare against the CRC-32 read from the zip local file header (Step 4).
Reject on mismatch.
Rejection: CRC mismatch → free the allocated region → go to Failure: zip format error.

Step 8 — Validate and mint: Mint.Lump(base, n)
Actor: Locator → Mint

Input: base, n (from Steps 5 and 4); lump binary resident at [base, base + 2^n × 4).

Action:

Call Mint.Lump(base, n). Mint performs its standard 9-step validation (see LumpFormat.md § "Mint Validation Sequence"):
Read Mem[base] — header word.
Verify magic[31:27] == 0x1F.
Verify n-6[26:23] <= 8; cross-check against transport-derived n.
Derive lumpSize = 2^(n-6+6).
Verify cw <= lumpSize - cc - 2.
Verify cc <= lumpSize - 2.
Scan freespace words — must all be zero.
Validate each c-list slot (well-formed GT Word 0).
Issue one E-GT; write Live NS slot.
On success, Mint writes the three NS words for object_id:
Word 1: base
Word 2: gt_seq[27:21] | limit_offset[20:0]
Word 3: spare[15] | G[1] | CRC-16[15:0] The NS slot transitions from Outform → Live.
Output: E-GT issued; NS slot is now Live.

Rejection: Any Mint validation step fails → Mint frees the allocated region → go to Failure: Mint rejection.

Step 9 — Locator returns; LOAD retries
Actor: Locator → Hardware

Input: Live NS slot for object_id (written by Mint in Step 8).

Action:

The Locator RETURNs to the same thread that invoked it.
The hardware automatically retries the LOAD instruction that fired the Absent event. The NS slot is now Live; LOAD resolves normally, populates the CR, and the calling thread continues execution.
Note: The calling thread's register state is fully preserved across the Absent event — from the thread's perspective, LOAD appears to have executed atomically (albeit with a latency cost).

Outform GT Preservation and Eviction
The Outform GT (typ=11 Word 0 in the caller's c-list slot) is not modified by the Absent event or the Locator. The c-list slot permanently holds typ=11 | object_id — the NS slot is the source of truth for whether the lump is currently resident.

The 96-bit IDE token saved in Step 2 allows eviction to restore the Outform state:

Eviction sequence (Locator.evict):
  1. Revoke all issued E-GTs for object_id
       → Mint increments gt_seq in NS Word 2
       → Any holder whose c-list Word 0 gt_seq mismatches will fault on next LOAD
  2. Free physical region [base, 2^n × 4 words)
       → Memory Manager.free(base)
  3. Restore NS slot to Outform state
       → Write saved 96-bit IDE token back into NS Words 1–3
  4. NS slot is now Outform again — next LOAD fires Absent event and
       re-triggers the full lazy-load protocol transparently
Any c-list holder (whether the original caller or any thread that received the E-GT) will trigger a new lazy-load on its next LOAD attempt after eviction.

Failure Modes
Failure: fetch error
Cause: Network error, timeout, or resource not found during Step 3.

State left:

No physical region allocated.
NS slot unchanged (still Outform, IDE token intact).
The Locator RETURNs a fault code to the thread.
The calling LOAD does not retry — it raises a fault to the thread's fault handler.
Recovery: The Outform NS slot is intact; a subsequent LOAD will re-trigger the Absent event and retry the full protocol.

Failure: zip format error
Cause: Bad signature, bit 3 set, invalid size, decompress error, or CRC-32 mismatch (Steps 4, 6, or 7).

State left:

Any allocated physical region is freed before returning.
NS slot unchanged (still Outform, IDE token intact).
The Locator RETURNs a fault code.
The calling LOAD raises a fault.
Recovery: Same as fetch error — Outform state is intact.

Failure: Mint rejection
Cause: Lump binary fails Mint's validation (Steps 1–8 of Mint.Lump): wrong magic, n mismatch, non-zero freespace, or malformed c-list slot.

State left:

Mint frees the allocated physical region (the region allocated in Step 5).
The NS slot is never written by Mint when validation fails — Mint writes the Live NS slot only on successful completion of all 9 validation steps. The NS slot therefore remains in its unchanged Outform state (IDE token intact in Words 1–3). No restore operation is needed; nothing was modified.
The Locator RETURNs a fault code.
The calling LOAD raises a fault.
Ownership summary: Mint owns physical region allocation/free only. NS slot ownership remains with the Locator throughout — Mint writes it exactly once on success and never touches it on failure.

Note: A persistent Mint rejection (same binary served every time) indicates a compiler error or a corrupted content-addressed cache. The IDE should invalidate the cache entry and re-publish the lump.
